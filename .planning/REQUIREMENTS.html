Requirements — Heimdall

Defined 2026-05-12 · Last updated 2026-05-19 (milestone v1.1 — JL-C1..JL-C9 added for LinkedIn Scraping by Company)
Core Value: Owner can run the entire executive job search from one place — track companies, log interactions, move applications through pipeline stages, and surface the highest-value introduction paths — without leaving the app.

Heimdall is brownfield. The Validated section captures the shipped surface area inferred from .planning/codebase/ and docs/summary.md. The v1 Active section captures the next-up work derived from bug.md, the "Needs Debugging" section of docs/summary.md, and .planning/codebase/CONCERNS.md.

v1.1 Active — LinkedIn Scraping by Company

Extends the v1 scrape-linkedin-connections skill (JL-B1..JL-B5) so it can scrape 2nd-degree connections at any target company, not just at companies attached to a specific LinkedIn job posting. Reuses the existing job_leads + prospects tables via "synthetic" leads where linkedinJobUrl is null.

Skill Input Parsing

  • JL-C1 · Skill accepts a LinkedIn company URL (https://(www.)?linkedin.com/company/<slug>(/.*)?) as a positional argument and uses it as the navigation entry point — distinct from the existing job-URL branch.
  • JL-C2 · Skill accepts a bare company-name string (not a UUID, not an https URL, not empty); the skill performs a LinkedIn company search and selects/disambiguates the target company before scraping.

Company-Scope Job Lead Creation

  • JL-C3 · The Heimdall API accepts a company-scope job-lead creation request and returns a job_leads row with linkedinJobUrl = null, roleTitle = null (or canonical "Company-wide scrape"), status = 'queued', FK-linked to a companies row (created on the fly if absent).
  • JL-C4 · The job_leads schema permits the company-scope row shape end-to-end — linkedinJobUrl and roleTitle are nullable; no constraints/indexes/types assume non-null.

Disambiguation UX

  • JL-C5 · On multi-match company names, the skill extracts the top 3–5 LinkedIn results (name + employees + industry), presents them inline as a numbered list, waits for the user to pick one, and uses the picked company's URL for the remaining scrape.

Navigation Branching

  • JL-C6 · Skill nav branches on the lead's linkedinJobUrl: null → navigation starts directly at /company/<slug>/people/; non-null → existing job → company → employees flow runs unchanged.

Drain Mode Integration

  • JL-C7 · Drain mode (no-arg invocation) processes company-scope leads via the same GET /api/job-leads?status=queued endpoint as job-URL leads and runs the linkedinJobUrl=null nav branch automatically — no separate queue, no separate status enum, no separate route.

UI for Company-Scope Leads

  • JL-C8 · The job-lead detail page renders gracefully when linkedinJobUrl is null — the "View job posting" affordance is hidden, a "Company scrape" label/badge replaces the role-title display, and the company name + employee count (once scraped) are shown prominently.
  • JL-C9 · The job-lead list view visually distinguishes company-scope leads from job-URL leads (different icon, label, or row badge) so the user can scan the queue without opening every lead.
