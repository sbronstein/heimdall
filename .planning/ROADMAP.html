Schema, Enums, and State Machine In Progress · 1/3 complete

The data model for outreach campaigns is in Postgres and the email status lifecycle is enforced at the API boundary before any other code depends on it.

  • canEmailTransition() at src/features/outreach/lib/email-status.ts rejects invalid moves and accepts valid ones (pending → generated → edited → approved → drafted) — pinned by Vitest
  • Drizzle-inferred types OutreachCampaign and OutreachEmail are exported from src/lib/domain/types.ts
  • ✓ 11-01 — Enums, two tables, barrel + migration 0013 applied to live Neon (Wave 1) — COMPLETE
  • ☐ 11-02 — Inferred types + enum value arrays + canEmailTransition() state machine (Wave 2)
  • ☐ 11-03 — Vitest state-machine test + PGlite schema-regression test (Wave 3)
Phase 12 · v1.2

API Routes Not started

The full REST API surface for outreach campaigns is live and usable from the CLI; skills and UI have working endpoints to build against.

CAMP-06, CAMP-07, CAMP-08, GEN-05
  • User (via CLI) can create a named campaign with a goal/instruction and get back the new campaign id
  • User (via CLI) can add a list of contact IDs to a campaign in one request; a second add of the same contact is silently deduped
  • User (via CLI) can list campaigns and see per-campaign counts: selected / generated / approved / drafted
  • User (via CLI) can transition an email's status through the lifecycle; invalid transitions return 400 with the state machine's rejection reason
  • All new routes return the standard { success, data, error, meta } envelope and reject unauthenticated requests
Phase 13 · v1.2

Triage Connection-Date Filter Not started UI

The owner can filter the existing triage queue by connection year or date range to surface cohorts of people (e.g. ID.me colleagues from 2021–2022).

TRGE-01
  • Year buttons in the triage UI filter the queue to contacts whose linkedinConnectionDate falls within that year
  • A two-year range (e.g. 2021–2022) can be set and the queue shows only matching contacts
  • The year filter survives a page reload (URL-driven via nuqs query params)
  • Clearing the filter restores the full triage queue
Phase 14 · v1.2

Campaign Builder UI Not started UI

The owner can create a named campaign with a goal by filtering contacts and multi-selecting recipients from the browser.

CAMP-01, CAMP-02, CAMP-03, CAMP-04, CAMP-05
  • Navigating to /dashboard/outreach/ shows the campaign list with per-campaign progress
  • The campaign builder lets the owner filter contacts simultaneously by howMet, connection year/date range, closeness tier, and outreach status
  • The owner can checkbox-select individual contacts or use select-all within the current filter
  • Providing a campaign name and goal and saving creates the campaign and navigates to its review page
  • Selecting the same contact twice across separate saves does not create a duplicate email row
Phase 15 · v1.2

Review and Approval UI Not started UI

The owner can review generated email content for each contact in a campaign, edit it inline, and approve or regenerate individual emails from the browser.

REV-01, REV-02, REV-03, REV-04, REV-06
  • The campaign review page shows all emails with their current status and, once generated, the subject line and body
  • The owner can edit a generated email's subject and body inline; saving the edit advances the status to edited
  • The owner can approve an email; the approve button is disabled for contacts with archived_at set
  • The owner can click regenerate on a single email to reset it to pending
  • Contacts with no stored email show a "needs LinkedIn message" badge; the campaign header shows progress counts (approved / total)
Phase 16 · v1.2

Email Generation Skill Not started

Running the generate-outreach-emails skill fills all pending emails in a campaign with personalized subject lines and bodies, in the owner's voice, without hallucinating history.

GEN-01, GEN-02, GEN-03, GEN-04
  • Running the skill against a campaign advances all pending emails to generated with a subject line and body written back via REST
  • Generated emails reference only facts present in the provided contact context; low-context contacts (fewer than 2 logged interactions) are flagged in the write-back
  • Generated tone is calibrated to closeness tier — conversational for tiers 1–2, professional-warm for tiers 3–5, brief and direct for tiers 7–8
  • Every generated email passes a built-in LLM-tell scan (no em-dashes, "leverage", "robust", or generic openers)
  • When generation fails for a contact, that email is marked failed and the skill continues without crashing
Phase 17 · v1.2

Gmail Drafting and Email Discovery Skill Not started Research needed

Approved emails are pushed to Gmail as drafts (never sent), contacts without stored emails have addresses discovered from Gmail thread history, and LinkedIn-only contacts are clearly flagged.

DISC-01, DISC-02, DISC-03, DRFT-01, DRFT-02, DRFT-03, DRFT-04, DRFT-05
  • Running the skill creates a Gmail draft for each approved email with a recipient; draft id stored, email marked drafted, contact's outreachStatus set to reached_out, timeline event logged
  • For approved emails where the contact has no stored email, the skill searches Gmail threads and writes back a discovered address only when at least two signals match; ambiguous multi-match candidates surface in the review UI
  • When no email is found, the contact's channel is set to linkedin_message and the badge appears in the review UI — never silently dropped
  • Re-running the skill is idempotent: no duplicate drafts; re-drafting an edited email updates the existing Gmail draft in-place
  • The skill contains zero send-family Gmail calls — every Gmail action is create_draft or drafts.update, verifiable by grepping for "send"
