Phase 2: Test Infrastructure — Context

Heimdall · gathered 2026-05-12 · Ready for planning

Phase Boundary

Stand up a Vitest test harness against the TypeScript codebase, pin the load-bearing logic surfaces with assertions, and add a regression test that fails if the BUG-01 hydration crash is reintroduced.

In scope

  • TEST-A1: Vitest + TypeScript wiring, PGlite-backed Drizzle test DB harness, npm test script.
  • TEST-A2: Envelope, canTransition(), logTimeline() side-effect, CSV parse, bridge-score, plus parseCursor/parseLimit and inferSeniority.
  • TEST-A3: BUG-01 regression pin — SSR structural test and hydration mount with console.error spy.

Out of scope (other phases)

  • Auth-gate test sweep across /api/* — Phase 3
  • match-connections.ts + scrapers — Phase 5
  • Playwright/E2E test runner
  • Coverage thresholds & CI test step
  • Coverage of routes/modules Phase 5 will rewrite

Implementation Decisions

Framework & Layout (carried from Phase 1)

  • D-01 Test runner: Vitest. No Jest, no node-tap.
  • D-02 Tests are colocated with source (pipeline.test.ts next to pipeline.ts). No top-level tests/ dir.
  • D-03 Naming: [file].test.ts (not .spec.ts).
  • D-04 Fixtures colocated: src/features/.../__fixtures__/....

DB Harness

  • D-05 Use PGlite (@electric-sql/pglite) as in-process Postgres. Real SQL, supports JSONB + text arrays.
  • D-06 Schema applied via Drizzle migrations from drizzle/migrations/. Planner picks wiring (per-file bootstrap vs. shared instance).
  • D-07 Do NOT mock logTimeline. Verify side-effect by querying real timeline_events rows in PGlite.
  • D-08 Pure-logic tests (canTransition, bridge-score, parseCursor, CSV parse, inferSeniority) don't need PGlite — plain in/out.

Test Environment Strategy

  • D-09 Single vitest.config.ts defaults to environment: 'node'. Hydration mount opts in via // @vitest-environment jsdom pragma.
  • D-10 One npm test command runs everything. No second config, no projects mode, no environmentMatchGlobs.

TEST-A2 Coverage Scope

  • D-11 Cover the 5 roadmap surfaces: envelope, canTransition, logTimeline, CSV parse, computeBridgeScore.
  • D-12 Plus parseCursor/parseLimit (filters.ts) and inferSeniority (seniority.ts) — cheap and flagged as critical-but-untested in TESTING.md.
  • D-13 Do NOT cover match-connections.ts, scraper modules, or linkedin-browser.ts — Phase 5 will rewrite parts of these.

TEST-A3 Regression Strategy

  • D-14 Two tests, one file. SSR structural (node): no <div> inside <button>, UserAvatarProfile markup present. Hydration mount (jsdom pragma): SSR-render, hydrate, spy on console.error for hydration warnings.
  • D-15 Hydration test relies on React warning text — acceptable flakiness; structural test is the deterministic backstop.
  • D-16 Mock @clerk/nextjs in the BUG-01 test; return fixed user object.

Existing Code Insights

Reusable Assets

  • drizzle/seed.ts patterns translate directly to test-fixture seeding inside PGlite.
  • src/lib/api/types.ts — exports both ApiResponse<T> type and the factory functions for response construction.
  • @faker-js/faker already in devDependencies.

Integration Points

  • Drizzle client singleton at src/lib/db/index.ts must be replaced with PGlite-backed instance before any API-route module imports it. Use vi.mock('@/lib/db') returning the PGlite-backed db.
  • Once the singleton is replaced, logTimeline writes to PGlite automatically — no separate mock.
  • API route handlers receive standard Request objects — call them directly with new Request(...). No Next.js test server needed.

What the Planner DOES Need to Research

  • PGlite + Drizzle wiring — exact API to apply migrations programmatically.
  • Whether next/server Request/Response types work in test handlers without polyfill.
  • Whether @clerk/nextjs mocking has import-time side effects.
