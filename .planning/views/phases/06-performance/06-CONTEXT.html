Phase 6: Performance — Context

Heimdall · gathered 2026-05-14 · Ready for planning · Mode: discuss

Phase Boundary

Eliminate per-row DB round-trips and add hot-path indexes so the 1500-contact dataset stops straining import, bulk categorize, prospect writes, bridge-building, and recommendations. Horizontal: N+1 elimination + index migration. Scope = the five PERF-A* requirements plus three incidentals discovered in scout plus restoring the matchConnections caller that Phase 5's reshape unintentionally orphaned.

Critical Findings From Codebase Scout

  • ROADMAP SC #1 first half already satisfied. Phase 5 deleted the in-app scraper; /api/job-leads/[id]/prospects/route.ts:60 already does db.insert(prospects).values(rows) in a single statement. SC text is stale.
  • match-connections.ts is currently dead code. Phase 5 removed its caller and didn't re-wire. prospectBridges never gets populated → recommendations + triage silently return empty results. PERF-A2 is meaningless until a caller is restored.
  • Bonus N+1 in recommendations/route.ts:44-52 — per-row db.update(prospectBridges).set({score}) in a for.
  • Bonus N+1 in /api/contacts/import:117 — per-row db.insert(contacts) in the CSV loop. Required co-fix for PERF-A5's ON-CONFLICT change.

In scope

  • PERF-A1 bridges half + PERF-A2: restore matchConnections caller inline in POST /prospects, wrapped in db.transaction(). Bulk bridge insert via .onConflictDoNothing().
  • PERF-A3: single UPDATE ... FROM unnest() via Drizzle's sql template tag.
  • PERF-A4: Drizzle index() on contacts(archived_at, linkedin_url, company_id, linkedin_connection_date) + companies(name) + UNIQUE partial on contacts(linkedin_url) WHERE NOT NULL.
  • PERF-A5 import half: bulk insert + ON CONFLICT on linkedin_url + narrowed SELECT for name+company secondary dedup.
  • PERF-A5 match-connections half: narrowed SELECT keyed on mutual-name tokens. Fuzzy match stays in Node.
  • Incidental fold: ROADMAP SC #1 wording refresh.
  • Incidental fold: /api/contacts/import for-loop → single bulk insert.
  • Incidental fold: recommendations/route.ts:44-52 bridge-score N+1 fixed.

Out of scope (deferred)

  • pg_trgm GIN indexes for ilike search across /api/search + list routes (v2).
  • Server-side fuzzy matching in match-connections.ts via pg_trgm similarity() or FTS.
  • Materialized view for recommendations.
  • CREATE INDEX CONCURRENTLY — overkill for single-user + 1500-row tables; plain CREATE INDEX wins.
  • Row-granular error reporting on /api/contacts/import — traded for aggregate counts.
  • Deleting matchConnections / retiring bridges feature.
  • New caching layers, query memoization.
  • Standalone btree on prospectBridges(prospect_id) / (contact_id) — planner research item.
  • prospectBridges.score column retirement.

Implementation Decisions

Bridge-Building Flow (PERF-A2 Caller Restoration)

  • D-01 — matchConnections called inline inside POST /api/job-leads/[id]/prospects, after bulk prospect insert, before status flip to 'found'.
  • D-02 — Whole handler wrapped in db.transaction(). Prospect insert + bridge insert + status update commit/rollback atomically. Failure → skill sees 500 → retry. Neon HTTP coalesces into one transactional request.
  • D-03 — Signature change minimal: pass scrapedProspects from the validated request body (already includes mutualConnectionNames). Planner decides whether to thread tx handle or wrap externally.
  • D-04 — Bulk bridge insert: await tx.insert(prospectBridges).values(bridgeValues).onConflictDoNothing(). Piggybacks on existing unique('prospect_bridge_unique').

Bulk Update Mechanism (PERF-A3)

  • D-05 — UPDATE ... FROM unnest() via sql template tag inside db.execute(). One round-trip; no N+1.
  • D-06 — CLAUDE.md "no raw SQL" guard respected — sql template tag is the documented Drizzle escape. One-line comment notes the exception.
  • D-07 — Response shape unchanged: { updated, total }.

DB-Side Dedup (PERF-A5)

  • D-08 — UNIQUE partial index on contacts(linkedin_url) WHERE linkedin_url IS NOT NULL. Doubles as the PERF-A4 btree on the same column. ON CONFLICT uses this partial index as its conflict target.
  • D-09 — Name+company dedup stays in-app, but uses a narrowed SELECT filtered by the CSV's tuples. No full table scan.
  • D-10 — /api/contacts/import becomes a single bulk insert with onConflictDoNothing({ target: contacts.linkedinUrl }). Per-row error messages drop to validation-stage only; database-level errors reported as a single aggregate.
  • D-11 — match-connections.ts narrowed SELECT: tokens drawn from scrapedProspects.flatMap(p => p.mutualConnectionNames). WHERE lower(first_name) IN (...) OR lower(last_name) IN (...). In-Node fuzzyMatch stays.

Index Migration (PERF-A4)

  • D-12 — Indexes declared in Drizzle schema via index('...').on(...). Plain CREATE INDEX (not CONCURRENTLY).
  • D-13 — 5 btree + 1 UNIQUE partial:
    1. contacts(archived_at)
    2. contacts(linkedin_url) UNIQUE WHERE NOT NULL — doubles as ON CONFLICT target
    3. contacts(company_id)
    4. contacts(linkedin_connection_date)
    5. companies(name)
  • D-14 — Migration 0008_*.sql from npm run db:generate. Single file. Applied via npm run db:migrate.

Recommendations N+1 (Incidental Fold #3)

  • D-15 — recommendations/route.ts:44-52 loop replaced by bulk UPDATE ... FROM unnest() for any bridges with score IS NULL. Alternative — compute on-the-fly without persistence — acceptable if planner verifies no other consumers.
