Critical Pre-Discovery

PD-01 · src/proxy.ts is dead code

Next.js convention requires the middleware file to be named middleware.ts at the project root or under src/. A repo-wide search confirms no such file exists; the only clerkMiddleware() invocation lives in src/proxy.ts, which Next.js does not auto-load. The "Clerk middleware enforces the single-user email lock" claim in PROJECT.md, ARCHITECTURE.md, and CLAUDE.md is currently inaccurate. /dashboard/* is enforced only by page-level auth() calls in src/app/page.tsx and src/app/dashboard/page.tsx.

Consequence: SEC-A1 is not "extend the middleware" — it is "activate the middleware for the first time, then expand it to /api/*." This is the single highest-leverage change in the phase.

PD-02 · Page-level redirects stay in place. src/app/page.tsx and src/app/dashboard/page.tsx already redirect unauthenticated users to /auth/sign-in. After Phase 3 these become belt-and-suspenders behind the now-active middleware; do NOT remove them in this phase.

Implementation Decisions

Enforcement Strategy

D-01 Rename src/proxy.ts → src/middleware.ts via git mv. Single-file change that activates everything.

D-02 Expand the protected-route matcher inside clerkMiddleware() from ['/dashboard(.*)'] to ['/dashboard(.*)', '/api/(.*)'].

D-03 Email lock applies to both route classes equally. CLI consumes the API surface using Steve's Clerk session; multi-user is out-of-scope.

D-04 For API routes, the planner must verify whether Clerk's auth.protect() emits a JSON 401 or an HTML redirect in Next.js 16. If it returns a redirect, add an explicit short-circuit emitting NextResponse.json({ success: false, error: 'Unauthorized' }, { status: 401 }) to preserve API-V1 envelope.

D-05 Public auth pages (/auth/sign-in, /auth/sign-up) remain reachable without a session. The protected list contains only [dashboard, api], so the negative top-level matcher already excludes them.

D-06 Do NOT add per-route auth() calls to src/app/api/**/route.ts. Middleware is the single chokepoint.

Verification Strategy (Phase 3 SC #1)

D-07 Two-layer verification, both required:

  1. Middleware unit test — src/middleware.test.ts: stubs Clerk via vi.mock('@clerk/nextjs/server') and asserts the middleware returns 401 / sign-in redirect for unauthenticated /api/* and /dashboard/* requests.
  2. Route-enumeration assertion — globs src/app/api/**/route.ts and asserts every API path matches the protected-route matcher pattern. Catches matcher drift.

D-08 Do NOT attempt a per-route HTTP sweep test — middleware runs at the edge before handlers, so a per-route test would need to mock middleware to reach the handler, which defeats the point.

D-09 Use the Phase 2 Vitest harness as-is (no PGlite needed for these tests).

Note: route count is 32, not 34. ROADMAP.md's figure was a slight overcount. The contract is "all /api/* routes," not a specific integer.

Starter-Template Auth Cleanup (SEC-A2)

D-10 Delete src/features/auth/components/github-auth-button.tsx outright. Remove the rendered <GithubSignInButton /> usage from sign-in-view.tsx and sign-up-view.tsx.

D-11 Remove the try { fetch('https://api.github.com/repos/...') } block from both src/app/auth/sign-in/[[...sign-in]]/page.tsx and src/app/auth/sign-up/[[...sign-up]]/page.tsx.

D-12 Remove the stars prop from SignInViewPage and SignUpViewPage, and delete the now-dead <Link>...Star on GitHub...{stars}</Link> block in both view components.

D-13 Do NOT touch other auth-page chrome (Random Dude quote, Logo SVG, grid pattern, top-right Login link). Phase 4 territory.
