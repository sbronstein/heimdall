Per D-23: Postgres `ALTER TYPE ADD VALUE` is non-transactional and CANNOT run inside an explicit `BEGIN; ... COMMIT;` block. Hand-edit the generated file ONLY if drizzle-kit wraps the statements in `BEGIN;` / `COMMIT;` (the 0002 analog confirms drizzle-kit does NOT wrap in Heimdall — but if a future drizzle-kit version starts wrapping, this is the only failure mode that requires intervention).

**Hand-edit fallback rules — IF and ONLY IF the generated file contains `BEGIN;` or `COMMIT;`:**
1. Strip the `BEGIN;` and `COMMIT;` lines so the ALTER TYPE statements run outside a transaction
2. Preserve any `AFTER '<value>'` or `BEFORE '<value>'` clauses on the `ADD VALUE` statements — both positional forms are valid and the migration semantics depend on the value's position in the enum order
3. Change the two `ALTER TYPE` statements to use `ADD VALUE IF NOT EXISTS` (idempotency guard for safe re-runs)
4. Ensure the two ALTER TYPE statements appear at the top of the file with `--> statement-breakpoint` between each statement

Do NOT strip `AFTER` / `BEFORE` clauses as a routine cleanup — those clauses are not the problem D-23 warns about. The actual problem D-23 names is non-transactional execution (BEGIN/COMMIT wrapping). Both `ADD VALUE 'queued' BEFORE 'searching'` and `ADD VALUE 'queued' AFTER 'scraped'` are valid Postgres and produce the same enum order; whichever form drizzle-kit emits should be left as-is.
