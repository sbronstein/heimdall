D-08 state machine (from CONTEXT.md):
- pending → scraping (existing job-page-scrape flow)
- scraping → scraped (existing) or scraping → pending (existing fallback on scrape error)
- scraped → queued (new: POST /search becomes thin flip)
- queued → searching (skill claims)
- searching → found (skill success — writes prospects)
- searching → failed (skill failure — writes lastError)
- failed → queued (retry — clears lastError)
- found → ready / actioned / archived (existing)
- ready → actioned / archived (existing)
- actioned → archived (existing)
- archived: terminal

Zod schema for bulk prospects (D-22, exact shape):
```typescript
const prospectSchema = z.object({
  name: z.string().min(1).max(200),
  title: z.string().max(300).nullable(),
  linkedinUrl: z.string().url().nullable(),
  mutualConnectionNames: z.array(z.string().max(200)).max(50)
});
const bulkBody = z.object({ prospects: z.array(prospectSchema).max(200) });
```

Response envelope helpers in src/lib/api/types.ts and src/lib/api/errors.ts:
- `success(data)` → 200, `created(data)` → 201, `paginated(data, meta)` → 200
- `notFound('Job lead')` → 404, `validationError(msg)` → 400, `serverError(err)` → 500

Timeline event types to emit:
- `job_lead_search_queued` — when PATCH /status or POST /search flips to 'queued'
- `job_lead_search_claimed` — when PATCH /status flips to 'searching'
- `job_lead_search_complete` — when POST /prospects flips to 'found' (existing event type — preserve)
- `job_lead_search_failed` — when PATCH /status flips to 'failed'

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Claude Code (skill) → POST /prospects | Untrusted body — the skill assembles prospect rows from LinkedIn DOM via agent-browser; malicious or malformed strings cross this boundary |
| Web UI / Skill → PATCH /status | Untrusted body — state-machine validation prevents fabricated terminal states |
| Web UI → POST /search | Untrusted (browser-supplied) lead id — UUID-shape enforced via Drizzle's `eq(jobLeads.id, id)` typed comparison; non-UUID inputs would fail at the DB layer with a 500 (acceptable — adversary supplies valid UUID-shaped input or gets a server error) |
