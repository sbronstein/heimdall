<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DB → React render | `lead.lastError` originates from the skill writing through PATCH /status; rendered as JSX text content in the failure banner |
| User input → clipboard API | `navigator.clipboard.writeText` writes a controlled string template — no user input flows into the clipboard contents |
| UI fetch → /api/job-leads/[id]/search | The retry button POSTs the lead id from current state — that id is server-vouched (came from RSC page query); no user-supplied id |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05-01 | Tampering | lastError reflected in UI (XSS surface) | mitigate | React JSX text content auto-escapes; no `dangerouslySetInnerHTML` used anywhere; lastError is capped to 220 chars at the API layer (Plan 04 Zod schema); split-on-colon is a string operation, not an HTML operation |
| T-05-05-02 | Information Disclosure | Clipboard content sniffing | accept | Single-user app; clipboard contains a benign command string (lead UUID + skill name); no secrets/tokens written |
| T-05-05-03 | Spoofing | Forged retry POST | mitigate | Browser path: Clerk middleware authenticates (Phase 3 + Plan 02 bypass irrelevant for browser); CLI/skill path: bearer token bypass (Plan 02) — both gates are enforced before the retry handler reaches the route |
| T-05-05-04 | Denial of Service | Tight-loop retry button click | accept | The retry handler is debounced by user click rate; the underlying API is sub-100ms per Plan 04 contract; YAGNI for explicit button disable-during-pending state |
</threat_model>
