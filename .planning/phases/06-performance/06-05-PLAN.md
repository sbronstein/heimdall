<decision_record>
**Decision (this plan): Variant B — drop persistence entirely.**

Evidence supporting Variant B over Variant A (bulk UPDATE):
1. `src/features/job-leads/lib/prioritization.ts:55` uses `bridge.score ?? computeBridgeScore(...)` — null scores are transparently recomputed by the downstream consumer.
2. The persisted `score` column is read by exactly TWO callsites:
   - `recommendations/route.ts:44-52` itself (the loop we are removing).
   - `prioritization.ts:55` (the fallback that makes Variant B safe).
   No other consumer relies on the persisted score.
3. `prospectBridges` has no `updated_at` column, so even a bulk UPDATE would not need timestamp management.
4. Variant B removes an entire write path from a GET handler, which is cleaner from a "pure read for GET" stance.
5. The persisted `score` column could be retired in a future cleanup (CONTEXT §Deferred §"prospectBridges.score column retirement"); Variant B is the natural lead-in.

Future re-evaluation triggers:
- If `computeBridgeScore` becomes expensive enough that repeated computation is observable (currently it's a single-expression arithmetic on 3 numbers, so this is not a concern).
- If a future feature needs to query "all bridges with score above N" — at that point either persist scores via a different path (e.g., a write-time hook in match-connections.ts) OR introduce a materialized view (out of scope).
</decision_record>
