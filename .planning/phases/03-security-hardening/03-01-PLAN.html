3. For `/dashboard/*` paths, preserve the existing `auth.protect()` behavior (HTML redirect to sign-in is correct for browser callers).

4. The email-lock check applies to both classes: after a session is confirmed present (via `userId`), check `sessionClaims?.email` and if non-empty and !== 'steve@bronstein.org', return `NextResponse.redirect(new URL('/auth/sign-in', req.url))` for BOTH path classes (D-03).

5. Keep the existing `export const config = { matcher: [...] }` block unchanged — the outer matcher already covers both `/api/*` and non-static dashboard paths; the gate logic happens via `isProtectedRoute` inside the callback per D-05.

Maintain the file's existing imports (`clerkMiddleware`, `createRouteMatcher`, `NextRequest`, `NextResponse`) — no new imports required. Use named identifiers only; no fenced code blocks elsewhere in the codebase need to change. Per D-06, do NOT add `auth()` calls to any `src/app/api/**/route.ts` file.
