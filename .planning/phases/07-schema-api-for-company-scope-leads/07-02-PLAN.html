Extend the existing `POST /api/job-leads` handler to accept a second body shape — `{ companyName, linkedinCompanyUrl? }` — that creates a company-scope job lead with `linkedinJobUrl: null`, `roleTitle: 'Company-wide scrape'`, and `status: 'queued'`. Resolve the company by case-insensitive name (auto-create stub on miss, backfill `linkedinUrl` on existing-match-with-null), enforce idempotent dedup against in-flight company-scope leads, and emit a `job_lead_created` timeline event tagged with `metadata.scope: 'company'`. Extend `src/app/api/job-leads/route.test.ts` with the five scenarios called out in CONTEXT §specifics.

Purpose: This route is the entire foundation Phases 8 (skill) and 9 (UI) depend on. The skill will POST to it; the UI will render the rows it creates. Without dedup, drain mode can stack identical leads on every retry; without auto-create, the user has to pre-curate a companies row before scraping; without the sentinel roleTitle, Phase 9 UI needs null-guards everywhere.

Output: Single API route handler accepts both job-URL and company-scope bodies; five new test scenarios pin the contract.
