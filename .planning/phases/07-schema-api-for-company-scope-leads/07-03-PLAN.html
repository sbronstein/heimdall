Pin the D-17 invariant: the existing `PATCH /api/job-leads/[id]/status` and `POST /api/job-leads/[id]/prospects` routes are input-shape agnostic — they operate on `id` lookups, never read or branch on `linkedinJobUrl`, and therefore must work identically against company-scope leads (`linkedinJobUrl: null`). No production code changes; pure regression tests that lock the behavior. Without these tests, a future refactor could add an accidental `if (lead.linkedinJobUrl)` guard that silently breaks the company-scope drain path — the skill's main loop assumes both routes work uniformly across lead shapes.

From src/app/api/job-leads/[id]/status/route.ts:

  • PATCH(request, { params }) — accepts { status: jobLeadStatusValues[number], lastError?: string }
  • Validates transition via canJobLeadTransition(currentStatus, newStatus)
  • On valid transition: updates jobLeads.status (+ lastError/lastErrorAt for failed), clears them for queued retries
  • Emits timeline event per transition: job_lead_search_queued, job_lead_search_claimed, job_lead_search_failed, etc.
  • Returns the full updated lead via success(updated)

From src/app/api/job-leads/[id]/prospects/route.ts:

  • POST(request, { params }) — accepts { prospects: Array<{ name, title, linkedinUrl, profileSnippet, mutualConnectionNames }> }
  • Requires lead.status === 'searching' (rejects with 400 + "Cannot write prospects to lead in status ''")
  • Bulk-inserts prospects, runs matchConnections inline, bulk-inserts bridges, flips status to 'found', emits 'job_lead_search_complete' timeline
  • All in a db.transaction() per Phase 6
